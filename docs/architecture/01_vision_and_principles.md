# 1. ビジョンと設計原則

## 1.1. アーキテクチャビジョン

### 目的
Surfin バッチフレームワークは、エンタープライズレベルのバッチ処理における「規律」と、Go言語の「機動力」を融合させることを目的とします。

### 目指す姿
「外側は堅牢な JSR-352、中身は Go の Goroutine エンジン」。
このハイブリッドな構造により、大規模データ処理におけるパフォーマンスと、エンタープライズ環境で求められる信頼性を両立させます。

---

## 1.2. 設計原則

Surfin Batch Frameworkは、以下の原則に基づいて設計されています。

1. **JSR-352 の尊重**: ジョブ、ステップ、チャンク、タスクレットといった標準的なバッチ処理の概念を継承し、予測可能性と保守性を担保します。
2. **Go の並行処理の活用**: Goroutine と Channel を活用したパイプライン処理により、I/O 待ちと CPU 処理を並列化し、スループットを最大化します。
3. **アダプターによる疎結合**: データベース、ストレージ、オブザーバビリティなどの外部システム連携は、抽象インターフェース（Adapter）を介して行い、技術スタックへの依存を最小化します。
4. **型安全性とジェネリクス**: Go のジェネリクスを活用し、コンパイル時の型チェックを徹底することで、ランタイムエラーを未然に防ぎます。
5. **責務の分離 (Separation of Concerns)**: ジョブ定義、実行エンジン、ビジネスロジック、永続化層を明確に分離します。
6. **依存性逆転の原則 (DIP)**: コアロジックは抽象インターフェースに依存し、具体的なインフラ実装はアダプター層で提供します。
7. **トランザクション境界の明確化**: ビジネスロジックからトランザクション管理を分離します。
