# Goバッチの思想は自前じゃなくていい

Goで書かれたバッチのトラブルを調査していたとき、ふと思った。**「SpringBatchだったら、これは起きなかっただろう」**

SpringBatchを書いていた頃には悩まなかった問題が、Goでは次々と現れた。なぜあのフレームワークは、こういう問題を起こさなかったのか。

そういえば、あの設計思想はJava標準仕様として策定されていたはずだ。それが **JSR-352** だった。久しぶりに仕様書を引っ張り出して読み返すうちに気づいた。**「フレームワークがやってくれていた」と思っていたことの正体が、ここに全部書いてあった。**

JSR-352はAccentureとSpringSourceが共同で策定した仕様だ。COBOLやメインフレームの時代から数十年にわたってバッチ処理と向き合ってきた実務経験が、この仕様の土台になっている。「解かれた問題」というより「何世代もかけて解き続けてきた問題」と言った方が正確かもしれない。

GoがDockerやKubernetesの成功とともにバッチ処理にも使われ始めたのは、ここ10年程度の話だ。バッチ設計の知見が数十年かけて体系化された一方で、Goエンジニアにはその歴史が継承されていない。同じ罠に再びはまっているのは、ある意味で必然だったのかもしれない。

## 1. Goバッチが運用フェーズで陥る罠

Goバッチで運用上の問題が起きやすい根本的な理由は、バッチ処理が本質的にステートフルだからだ。「どこまで処理したか」という進行状況を常に意識しなければならず、それを適切に管理する仕組みがないと、あらゆる問題の温床になる。

多くのGo製バッチが運用フェーズで陥る罠は、この「進行状況の管理」を現場の知恵（場当たり的な実装）に委ねてしまうことだ。

- **再実行のたびに0件から処理し直す**: チェックポイントがないため、障害発生時に最初からやり直すしかない。
- **失敗した位置を手動で特定してフラグを立てる**: DBのカラムに処理済みフラグを持たせ、インデックスを圧迫し、最終的にデッドロックの原因になる。
- **リトライのたびに重複データが生まれる**: トランザクションの境界が曖昧なため、再実行のたびに整合性の確認を人間が担う。

これらはGoが悪いわけではない。バッチ処理の「運用設計」が、コードの構造から分離されていないことに起因する。ではその「運用設計」とは何か。JSR-352はそれを明快に答えてくれていた。

## 2. JSR-352が提示していた「普遍的な設計原則」

JSR-352を読み込むと、そこには言語に依存しない、バッチ処理の「構成要素」が整理されている。

- **Chunk Processing（チャンク処理）**: データをまとめて読み込み、まとめて書き出す。この単位でトランザクションを区切り、チェックポイントを打つ。
- **JobRepository**: ジョブやステップの実行履歴だけでなく、再開に必要な実行状態（Execution Context）も永続化する管理層。障害発生時に「どこから再開するか」を支える中核となる。
- **Checkpoint**: 障害発生時に、0件目からではなく「前回成功した直後」から処理を再開するための防波堤。

重要なのは、これらがJava固有の魔法ではないということだ。

*「チャンク単位で区切る」「状態を外部に永続化する」「再開点を明示する」——この三原則は、どの言語でも実現できる論理構成だ。**

## 3. JSR-352の三原則で、何が解決されるのか

冒頭で挙げた三つの苦労が、JSR-352の設計原則によってどう対応されるか整理する。

**「再実行のたびに0件から処理し直す」→ Checkpointで解決**

チャンク単位で進捗を永続化しておけば、障害発生時に前回成功した直後から再開できる。0件からやり直す必要はない。

**「失敗した位置を手動で特定してフラグを立てる」→ JobRepositoryで解決**

処理状態を一元管理する層があれば、人間が手動でフラグを立てる必要がない。どこで止まったかはフレームワークが把握している。

**「リトライのたびに重複データが生まれる」→ Chunk Processingでトランザクション境界が明確になる**

Chunk Processingは冪等性そのものを保証するわけではない。ただ、トランザクションの境界をチャンク単位で明示することで、重複書き込みを防ぐ設計を実現しやすくなる。

これらの多くは、すでに解かれた問題だ。ではこの思想を、Goでどう実現するか。

## 4. GoでJSR-352の思想を実現する

### 責務の分離：ロジックと運用の切り分け

まず、データ加工のロジック（ビジネスロジック）と、運用ロジックを厳格に分離する。

```go
// 悪い例：ロジックの中に運用コードが混ざる
func ProcessBatch() {
    for _, item := range items {
        // 更新処理...
        // 失敗したらフラグを立てて...
        // 次回はこのフラグを見てスキップして...
    }
}

// 良い例：運用責任をランナーに委譲する
func (j *Job) Execute(ctx context.Context) error {
    return j.runner.Run(ctx, func(chunk []Data) error {
        // ここには「どう加工するか」だけを書く
        return processLogic(chunk)
    })
}
```

この構造により、開発者は「53万件目で止まった時にどうなるか」という運用上の不安から解放され、「1件をどう正しく加工するか」という本来の目的に集中できる。

### チェックポイントをコードで表現する

Goのインターフェースを使うと、チェックポイントの永続化をビジネスロジックの外側に自然に置ける。たとえばこういう構造だ。

```go
func runChunk(ctx context.Context, offset int, checkpoint Checkpoint) error {
    records, err := reader.ReadChunk(offset, chunkSize)
    if err != nil {
        return err
    }

    if err := processor.Process(records); err != nil {
        return err
    }

    // チャンク単位でチェックポイントを永続化する
    return checkpoint.Save(ctx, offset+len(records))
}
```

ビジネスロジックはProcessの中だけに閉じ込められ、「どこまで処理したか」はcheckpoint.Saveが責任を持つ。この分離がJSR-352の言っていたことそのものだった。

## 5. 思想はわかった。しかし実装は泥臭い

思想はわかった。コードのイメージもできた。しかし現実には、すでに動いている既存バッチにこの思想を後から組み込む場面の方が多い。

現場で運用されているバッチに「チェックポイント機能」や「リトライ制御」を後から加えることは、稼働中のシステムの根幹に手を入れる作業だ。トランザクションの境界はどこか、失敗したときの整合性は誰が保証するのか、リスタート時に重複して書き込まれるデータはないか——。これらを一から構築しようとすれば、本来書くべき「業務ロジック」よりも運用インフラのためのコードが肥大化する。

だからこそ、この設計原則を「コードとして再利用できる形」に落とし込んだランタイムやライブラリを活用するアプローチが有効になる。自前で書くにせよ既存のものを使うにせよ、重要なのは「運用の責務をビジネスロジックから分離する」という構造を、チームの共通認識として持つことだ。

（参考までに、私自身はこの設計を具体化する試みとして **[Surfin](https://github.com/tigerroll/surfin)** というGo製のバッチランタイムを開発している。興味があればその実装も一つのヒントになるかもしれない。）

## 6. GoとJSR-352の思想はなぜ相性がいいのか

JSR-352を読み返しながら、Goのコードが頭に浮かんだ。「あ、これGoで自然に書ける」と思った瞬間が何度もあった。

特に確信したのは、引き継いだGoバッチのトラブル対応中のことだ。バッチが失敗した夜、ログとDBのレコードを見比べながら「どこまで処理したんだっけ？」と頭を抱えていた。翌日、コードを書き直していたとき、最初は単純にProcessorを呼び出すだけのシンプルな構造だった。しかし「障害時にリスタートしたい」という要求が出た瞬間、チェックポイントの処理をロジックの中に書き込み始め、コードが急激に汚れていった。

そこで試したのが、Processorを外側から包む構造だ。

```go
type CheckpointProcessor struct {
    inner      Processor
    repository JobRepository
}

func (p *CheckpointProcessor) Process(ctx context.Context, item interface{}) error {
    // 1. 前処理（チェックポイントの確認）
    // 2. 本丸の処理
    err := p.inner.Process(ctx, item)
    // 3. 後処理（チャンク単位での永続化・オフセット更新）
    return p.repository.Save(ctx, ...)
}
```

そのとき気づいた。**「業務ロジック（inner.Process）は、自分がバッチ処理の一部であることすら知らない」**。業務ロジックの行数は1行も増えていないのに、運用上の安全装置だけが外側から綺麗に装着されている。

その瞬間、JSR-352の仕様書を読み返して背筋が凍った。そこに書かれていたJobRepositoryやChunkの概念は、私がGoのinterfaceを使って組み立てた構造そのものだったからだ。

Goのinterfaceはこういう「外側から包む」設計をとても自然に書かせてくれる。チェックポイントだけでなく、リトライやログも同じように外側から差し込める。必要に応じてチャンク単位の並列処理へ拡張することも容易だ。さらに、チェックポイントを外部ストレージに永続化する設計では、ジョブ本体は軽量でどこでも動ける方が望ましい。Goのシングルバイナリはその要件にも自然にフィットする。

JSR-352の思想は言語を選ばない。ただ、Goの設計とは特に自然に噛み合うと感じている。

もちろん、AirflowやTemporalのようなワークフローエンジンを使えば、こうした課題を別のレイヤーで解決することもできる。ただ、それらのツールを導入する前に「設計思想として理解しているかどうか」は、トラブルが起きたときの対応力に大きく影響すると感じている。ツールに頼る前に、思想を持つ。JSR-352はそのための手がかりになった。

## おわりに

Goバッチで苦労していたあの問題は、自分が思っていたよりずっと前に、すでに体系化されていた。

JSR-352を細部まで読み込む必要はないかもしれない。ただ、「チャンク単位で区切る」「状態を外部に永続化する」「再開点を明示する」という三つの考え方を知るだけでも、バッチの設計は大きく変わる。

Goには「シンプルに作る」という文化がある。私もその価値には強く共感している。一方で、バッチ処理は何十年にもわたって実運用の中で磨かれ、失敗と改善を繰り返しながら積み上げられてきた分野でもある。

だからこそ、実装をシンプルにすることと、過去の知見まで捨てることは違う。

Goでバッチを書くことは、ゼロから戦うことではない。実装は自前でもいい。ランタイムを使ってもいい。しかし、何十年もかけて解かれてきた問題まで、自前で解き直す必要はない。

私がJSR-352から学んだのは、フレームワークの使い方ではない。バッチ処理には、言語を超えて受け継ぐべき設計思想があるということだった。

その思想を借りるだけでも、Goバッチの設計はきっと変わる。そして、それはこれからGoでバッチを書く人たちが、同じ失敗を繰り返さないための一歩になると私は思っている。
