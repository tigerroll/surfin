# Goバッチの思想は自前じゃなくていい

Go で書かれたバッチのトラブルを調査していたとき、ふと思った。 **「Spring Batch だったら、これは起きなかっただろう」**

Spring Batch を書いていた頃には悩まなかった問題が、Go では次々と現れた。なぜあのフレームワークは、こういう問題を起こさなかったのか。

そういえば、あの設計思想は Java 標準仕様として策定されていたはずだ。それが JSR-352 だった。久しぶりに仕様書を引っ張り出して読み返すうちに気づいた。 **「フレームワークがやってくれていた」と思っていたことの正体が、ここに全部書いてあった。**

JSR-352（Java Batch）は、SpringSource や Accenture が起草し、JBoss コミュニティらが策定を推進した標準仕様だ。その土台には、COBOL の時代から数十年にわたり、先人たちがメインフレーム等の環境で試行錯誤しながらバッチ運用の定石を築き上げてきた歴史がある。

リスタート性の確保やチャンク単位のトランザクション管理といった、私たちが Go で直面する困難の多くは、実は彼らが標準化という枠組みの中で既に抽象化を試みたものだ。この設計の骨組みを読み解き、Go の作法で再構築することは、車輪の再発明を避け、過去の技術的蓄積を現代のアーキテクチャに継承するための最短距離となるのではないだろうか。

Go が Docker や Kubernetes の普及とともにバッチ処理にも使われ始めたのは、ここ 10 年程度の話だ。バッチ設計の知見が数十年かけて体系化された一方で、Go エンジニアにはその歴史が継承されていない。同じ罠に再びはまっているのは、ある意味で必然だったのかもしれない。

## 1. Go バッチが運用フェーズで陥る罠

Go バッチで運用上の問題が起きやすい根本的な理由は、バッチ処理が本質的にステートフルだからだ。「どこまで処理したか」という進行状況を常に意識しなければならず、それを適切に管理する仕組みがないと、あらゆる問題の温床になる。

多くの Go 製バッチが運用フェーズで陥る罠は、この「進行状況の管理」を現場の知恵（場当たり的な実装）に委ねてしまうことだ。

* **再実行のたびに 0 件から処理し直す**: チェックポイントがないため、障害発生時に最初からやり直すしかない。
* **失敗した位置を手動で特定してフラグを立てる**: DB のカラムに処理済みフラグを持たせ、インデックスを圧迫し、最終的にデッドロックの原因になる。
* **リトライのたびに重複データが生まれる**: トランザクションの境界が曖昧なため、再実行のたびに整合性の確認を人間が担う。

これらは Go が悪いわけではない。バッチ処理の「運用設計」が、コードの構造から分離されていないことに起因する。ではその「運用設計」とは何か。JSR-352 はそれを明快に答えてくれていた。

## 2. JSR-352 が提示していた「普遍的な設計原則」

JSR-352 を読み込むと、そこには言語に依存しない、バッチ処理の「構成要素」が整理されている。

* **Chunk Processing（チャンク処理）**: データをまとめて読み込み、まとめて書き出す。この単位でトランザクションを区切り、チェックポイントを打つ。
* **JobRepository**: ジョブやステップの実行履歴だけでなく、再開に必要な実行状態（Execution Context）も永続化する管理層。障害発生時に「どこから再開するか」を支える中核となる。
* **Checkpoint**: 障害発生時に、0 件目からではなく「前回成功した直後」から処理を再開するための防波堤。

重要なのは、これらが Java 固有の魔法ではないということだ。

**「チャンク単位で区切る」「状態を外部に永続化する」「再開点を明示する」——この三原則は、どの言語でも実現できる論理構成だ。**

## 3. JSR-352 の三原則で、何が解決されるのか

冒頭で挙げた三つの苦労が、JSR-352 の設計原則によってどう対応されるか整理する。

**「再実行のたびに 0 件から処理し直す」→ Checkpoint で解決**

チャンク単位で進捗を永続化しておけば、障害発生時に前回成功した直後から再開できる。0 件からやり直す必要はない。

**「失敗した位置を手動で特定してフラグを立てる」→ JobRepository で解決**

処理状態を一元管理する層があれば、人間が手動でフラグを立てる必要がない。どこで止まったかはフレームワークが把握している。

**「リトライのたびに重複データが生まれる」→ Chunk Processing でトランザクション境界が明確になる**

Chunk Processing は冪等性そのものを保証するわけではない。ただ、トランザクションの境界をチャンク単位で明示することで、重複書き込みを防ぐ設計を実現しやすくなる。

これらの多くは、すでに解かれた問題だ。ではこの思想を、Go でどう実現するか。

## 4. Go で JSR-352 の思想を実現する

### 責務の分離：ロジックと運用の切り分け

まず、データ加工のロジック（ビジネスロジック）と、運用ロジックを厳格に分離する。

```go
// 悪い例：ロジックの中に運用コードが混ざる
func ProcessBatch() {
    for _, item := range items {
        // 更新処理...
        // 失敗したらフラグを立てて...
        // 次回はこのフラグを見てスキップして...
    }
}

// 良い例：運用責任をランナーに委譲する
func (j *Job) Execute(ctx context.Context) error {
    return j.runner.Run(ctx, func(chunk []Data) error {
        // ここには「どう加工するか」だけを書く
        return processLogic(chunk)
    })
}

```

この構造により、開発者は「53 万件目で止まった時にどうなるか」という運用上の不安から解放され、「1 件をどう正しく加工するか」という本来の目的に集中できる。

### チェックポイントをコードで表現する

Go のインターフェースを使うと、チェックポイントの永続化をビジネスロジックの外側に自然に置ける。たとえばこういう構造だ。

```go
func runChunk(ctx context.Context, offset int, checkpoint Checkpoint) error {
    records, err := reader.ReadChunk(offset, chunkSize)
    if err != nil {
        return err
    }

    if err := processor.Process(records); err != nil {
        return err
    }

    // チャンク単位でチェックポイントを永続化する
    return checkpoint.Save(ctx, offset+len(records))
}

```

ビジネスロジックは Process の中だけに閉じ込められ、「どこまで処理したか」は checkpoint.Save が責任を持つ。この分離が JSR-352 の言っていたことそのものだった。

## 5. Go と JSR-352 の思想はなぜ相性がいいのか

JSR-352 を読み返しながら、ふと Go のコードが思い浮かんだ。「あ、これ Go で書ける」という確信だ。

特に思い出すのは、引き継いだ Go バッチのトラブル対応中だ。バッチが失敗した夜、ログと DB のレコードを見比べながら頭を抱えていた。翌日、コードを書き直す中で「障害時にリスタートしたい」という要求を満たそうとすると、チェックポイントの処理がロジックの中に混入し、コードが急激に汚れていくことに気づいた。そこで試したのが、Processor を外側から包む構造だ。

```go
type CheckpointProcessor struct {
    inner      Processor
    repository JobRepository
}

func (p *CheckpointProcessor) Process(ctx context.Context, item interface{}) error {
    // 1. 前処理（チェックポイントの確認）
    // 2. 本丸の処理
    err := p.inner.Process(ctx, item)
    // 3. 後処理（チャンク単位での永続化・オフセット更新）
    return p.repository.Save(ctx, ...)
}

```

この構造に気づいたとき、仕様書を読み返して思わず息を呑んだ。そこに書かれていた **JobRepository** や **Chunk** といった概念は、私が interface で組み立てた構造そのものだったからだ。私が模索していた Go の設計は、数十年前から標準仕様として定義されていたのだ。まるで自分が描いていた設計図を、先に誰かが完成させていたような答え合わせの衝撃を受けた。

なぜ、この JSR-352 の思想と Go はこれほどまでに相性がいいのか。理由は明確だ。「Go の interface が、運用要件とビジネスロジックを分離するのに最適だから」に他ならない。

**Go と JSR-352 の相性がいい 3 つの理由**

1. **「疎結合なデコレーション」が標準であること**

* Go の interface は、実装の詳細を知らなくても「Processor（処理できるもの）」として包み込める。これにより、業務ロジックは自分がバッチの一部であることすら知らずに、運用上の安全装置を外側から綺麗に装着できる。これは JSR-352 が目指した「ロジックと運用ランタイムの分離」を、言語機能として最小限の記述で実現できることを意味する。

2. **「シングルバイナリ」と「分散バッチ」の親和性**

* JSR-352 の設計では、JobRepository を外部に置くことで、ジョブ本体は軽量でどこでも動けることが求められる。Go のシングルバイナリは、Kubernetes の Job などで立ち上げれば即座にバッチとして動作する。この「運用の責務（Repository）」と「処理の責務（Job）」の物理的な分離と相性が極めていい。

3. **複雑な抽象化を嫌う文化**

* Java のフレームワークが巨大な継承やアノテーションで隠蔽しようとしたものを、Go は interface とコンポジションでシンプルに書き下せる。JSR-352 という「過去の巨大な仕様」を、Go であれば「数行のインターフェース」として再定義できる。思想は過去の巨人に学びつつ、実装は Go の文化に従ってクリーンに保てるのだ。

JSR-352 の思想は言語を選ばない。ただ、Go の言語機能と運用文化にとって、この設計原則はあまりにも自然な選択肢だったのだと感じている。

## 6. 思想は理解した。だが、現実は一筋縄ではいかない。

思想は理解した。コードのイメージもできた。しかし現実には、すでに動いている既存バッチにこの思想を後から組み込む場面の方が多い。

現場で運用されているバッチに「チェックポイント機能」や「リトライ制御」を後から加えることは、稼働中のシステムの根幹に手を入れる作業だ。トランザクションの境界はどこか、失敗したときの整合性は誰が保証するのか、リスタート時に重複して書き込まれるデータはないか——。これらを一から構築しようとすれば、本来書くべき「業務ロジック」よりも、運用インフラのためのコードが肥大化してしまう。

だからこそ、この設計原則を「コードとして再利用できる形」に落とし込んだランタイムやライブラリを活用するアプローチが有効になる。自前で実装するにせよ、既存のものを使うにせよ、重要なのは「運用の責務をビジネスロジックから分離する」という構造を、チームの共通認識として持つことだ。

（参考までに、私自身はこの設計を具体化する試みとして **[Surfin](https://github.com/tigerroll/surfin)** という Go 製のバッチランタイムを開発している。興味があればその実装も一つのヒントになるかもしれない。）

## おわりに

Go バッチで苦労していたあの問題は、自分が思っていたよりずっと前に、すでに体系化されていた。

JSR-352 を細部まで読み込む必要はないかもしれない。ただ、「チャンク単位で区切る」「状態を外部に永続化する」「再開点を明示する」という三つの考え方を知るだけでも、バッチの設計は大きく変わる。

Go には「シンプルに作る」という文化がある。私もその価値には強く共感している。一方で、バッチ処理は何十年にもわたって実運用の中で磨かれ、失敗と改善を繰り返しながら積み上げられてきた分野でもある。

だからこそ、実装をシンプルにすることと、過去の知見まで捨てることは違う。

Go でバッチを書くことは、ゼロから戦うことではない。実装は自前でもいい。ランタイムを使ってもいい。しかし、何十年もかけて解かれてきた問題まで、自前で解き直す必要はない。

私が JSR-352 から学んだのは、フレームワークの使い方ではない。バッチ処理には、言語を超えて受け継ぐべき設計思想があるということだった。

その思想を借りるだけでも、Go バッチの設計はきっと変わる。そして、それはこれから Go でバッチを書く人たちが、同じ失敗を繰り返さないための一歩になると私は思っている。
