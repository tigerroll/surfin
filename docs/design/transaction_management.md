# トランザクション管理の設計

## 1. 概要
本ドキュメントは、Surfin バッチフレームワークにおけるトランザクション管理の設計方針を記述します。バッチ処理において、データベースの整合性を保ちつつ、ストレージなどの非データベースリソースとの連携を柔軟に行うための仕組みを定義します。

## 2. 設計方針
*   **コンテキストベースの伝播**: トランザクションは `context.Context` を介して伝播させます。これにより、`ItemWriter` などのコンポーネントがトランザクションの存在を意識しつつも、インターフェースをシンプルに保つことができます。
*   **抽象化されたトランザクションインターフェース**: `tx.Tx` インターフェースを定義し、具体的なデータベース実装（GORMなど）を隠蔽します。
*   **リソースの分離**: データベーストランザクションと、ストレージ操作などの非トランザクションリソースを明確に分離し、必要に応じて調整します。

## 3. 実装パターン

### 3.1. トランザクションの開始と伝播
`ChunkStep` などの実行エンジンは、チャンク処理の開始時にトランザクションを開始します。

1.  `tx.Begin()` を呼び出し、`tx.Tx` オブジェクトを取得。
2.  `tx.ContextWithTx(ctx, tx)` を使用して、コンテキストにトランザクションを格納。
3.  このコンテキストを `ItemWriter.Write(ctx, items)` に渡す。

### 3.2. ItemWriter での利用
`ItemWriter` の実装は、コンテキストからトランザクションを取得して利用します。

```go
func (w *MyWriter) Write(ctx context.Context, items []MyItem) error {
    tx, ok := tx.TxFromContext(ctx)
    if !ok {
        return fmt.Errorf("transaction not found in context")
    }
    // tx を使用してデータベース操作を実行
    return tx.Exec(...)
}
```

## 4. 非データベースリソースの扱い
ストレージ（S3, GCSなど）への書き込みは、通常データベースのようなACIDトランザクションをサポートしません。そのため、以下の戦略をとります。

*   **最終確定（Commit）の遅延**: 書き込み処理はバッファリングし、`Close` メソッドまたは `Flush` メソッドが呼ばれたタイミングで一括してストレージへアップロードします。
*   **アトミック性の確保**: 必要に応じて、一時ファイルへの書き込みとリネーム操作（アトミックなファイルシステム操作）を利用して、部分的な書き込みを防ぎます。

## 5. リソース管理とトランザクション処理の柔軟性向上戦略

### 5.1. リソース管理の汎用化
フレームワークの基盤となるリソース管理を汎用化し、データベース以外のリソース（例: ストレージ）への対応を容易にします。

### 5.2. ItemWriter のトランザクション処理の汎用化
`ItemWriter` インターフェースをデータベーストランザクションに直接依存しないように汎用化し、ストレージなどの非データベース書き込み処理にも対応できるようにしました。トランザクションは `context.Context` を介して渡されるように変更されました。

### 5.3. 実装のポイント
- `tx.TxFromContext(ctx)` を使用して、コンテキストからトランザクションを取得するパターンを標準化します。
- `JobFactory` が汎用的な `ResourceProvider` のマップを管理するようにし、拡張性を向上させます。
