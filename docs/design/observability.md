# 可観測性 (Observability) 設計

本ドキュメントは、Surfin バッチフレームワークにおける可観測性（メトリクス、トレース、ログ）の設計と実装戦略を統合したものです。

## 1. 抽象設計
Surfin フレームワークは、メトリクス、トレース、ログといった「オブザーバビリティ」の確保を不可欠としています。OpenTelemetry を標準として採用し、ベンダーニュートラルな連携を目指します。
- **ベンダーニュートラル**: 特定のバックエンドに依存しない OTLP (OpenTelemetry Protocol) を標準とする。
- **インターフェース分離**: `MetricRecorder` や `Tracer` インターフェースを介して、コアロジックと実装を分離する。
- **設定駆動**: `application.yaml` により、エクスポート先やプロトコルを動的に切り替え可能にする。

## 2. Metrics Adapter 設計
`metrics.MetricRecorder` インターフェースを介して、様々なオブザーバビリティバックエンドへのデータ収集、処理、エクスポートを可能にする抽象化レイヤーです。

### 2.1. OpenTelemetry Metrics Adapter
`pkg/batch/adapter/metrics/opentelemetry` に実装されるアダプターです。
- **目的**: `pkg/batch/core/metrics.MetricRecorder` インターフェースを OpenTelemetry Go SDK を用いて具体的に実装し、Surfin が生成するメトリクスを OTLP 互換の様々なバックエンドへエクスポート可能にします。
- **設計思想**:
    - **動的エクスポート**: 設定された各エクスポーターに対して、OpenTelemetry エクスポーターインスタンスを動的に作成します。
    - **属性付与**: ジョブ名、ステップ名、ID、ステータスなどのコンテキスト情報を属性としてメトリクスに付与します。
    - **ライフサイクル管理**: Fx モジュールを通じて、アプリケーションのシャットダウン時に適切にシャットダウン処理を行います。

## 3. OpenTelemetry Logs Adapter 設計
`slog` のログレコードを OpenTelemetry のログデータモデルに準拠した形式に変換し、OTLP を介して様々なオブザーバビリティプラットフォームに連携します。
- **実装**: `slog` ハンドラーを実装し、ログレコードを OTLP 形式に変換する。
- **マッピング**: ログレベル、タイムスタンプ、属性を適切にマッピングする。

## 4. ロギング設計 (slog 移行)

### 4.1. 目的と背景
従来のロギング実装は、Go標準の `log` パッケージをラップした非構造化ログを出力しており、Cloud Logging 等での高度なフィルタリングや集計が困難でした。これを解決するため、Go 1.21 で導入された `log/slog` へ移行しました。

### 4.2. 設計戦略
*   **構造化ログの導入**: JSON形式の構造化ログを出力し、ログ分析基盤での検索・フィルタリングを容易にします。
*   **既存APIの維持**: `pkg/batch/support/util/logger` の公開API（`Debugf`, `Infof` 等）は変更せず、内部実装のみを `slog` に置き換えることで、呼び出し元への影響を最小限に抑えます。
*   **設定による動的切り替え**: `application.yaml` (`surfin.system.logging.format`) により、JSON形式とText形式を動的に切り替え可能にします。アプリケーション起動時に `logger.SetLogFormat` を呼び出すことで、実行時にフォーマットを適用します。

## 5. 可観測性移行戦略
*   **ディレクトリ構造**: `pkg/batch/adapter/observability` 配下にメトリクスとトレースを統合。
*   **設定統合**: `surfin.adapter.observability` セクションに設定を集約。
*   **移行フェーズ**:
    1.  **基盤構築**: `observability` 配下にディレクトリを再編成し、共通設定を導入。
    2.  **設定統合**: `application.yaml` の設定を新しい共通構造に移行。
    3.  **統合とクリーンアップ**: Fx アプリケーションに組み込み、古いメトリクス関連コードを削除。
*   **実装**: OpenTelemetry を標準とし、ベンダーニュートラルな連携を実現する。
